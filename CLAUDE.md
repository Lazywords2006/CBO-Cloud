## Key Architecture Concepts

### BCBO Six-Phase Structure

The BCBO algorithm operates in 6 distinct phases with different search characteristics:

1. **dynamic_search** (10% of iterations) - Weak search, high exploration
2. **static_search** (10% of iterations) - Weak search, prone to stagnation
3. **encircle_dynamic** (25% of iterations) - Medium search, gradual convergence
4. **encircle_static** (20% of iterations) - Medium search, stable
5. **attack_dynamic** (20% of iterations) - Strong search, fast convergence
6. **attack_static** (15% of iterations) - Strong search, precision optimization

**Critical Design Principle**: The first two phases (dynamic_search, static_search) are intentionally weak and represent 20% of iterations. This is where hybrid algorithms like BCBO-DE insert enhancement mechanisms.

### BCBO-DE Embedded Fusion Strategy

The BCBO-DE fusion uses an **embedded hybrid approach** (inspired by MSA-DE paper, Elaziz 2019):

**Fusion Philosophy**:
- DO NOT replace BCBO entirely or create a new algorithm
- DO insert DE operators into weak BCBO phases only (dynamic_search, static_search)
- DO preserve BCBO's strong phases (encircle and attack phases)

**Fusion Mechanism**:
1. During fusion phases, split population by fitness
2. Top 70% â†’ Use original BCBO operators
3. Bottom 30% â†’ Use DE operators (mutation â†’ crossover â†’ selection)
4. Merge populations and continue to next iteration

**Adaptive Population Splitting**:
- Base ratio: 70% BCBO, 30% DE
- Adjusts based on population diversity:
  - Low diversity (< 0.3) â†’ Increase DE ratio to 50% (more exploration)
  - High diversity (> 0.7) â†’ Increase BCBO ratio to 80% (faster convergence)
  - Normal diversity â†’ Use default 70/30 split

### Cloud Task Scheduling Problem

**Problem Definition**: Assign M tasks to N virtual machines to minimize makespan and cost

**Task Properties** (7 dimensions):
- Length (MI), file size, output size, deadline
- Priority, cost factor, energy factor

**VM Properties** (7 dimensions):
- Processing speed (MIPS), memory, bandwidth
- Unit cost, energy consumption, CPU cores, status

**Fitness Function**: Multi-objective with primary focus on makespan
- Primary: Makespan (completion time) - heavily weighted
- Secondary: Load balancing, resource utilization, total cost
- Penalties: Energy consumption, SLA violations, resource violations

### Fitness Caching

The BCBO implementation uses fitness caching to avoid redundant evaluations:
- Cache key: tuple of task-to-VM assignments
- Significant performance improvement for large populations
- Clear cache when problem parameters change

### Adaptive Parameter Control

**DE Scaling Factor (F)** - Uses HDE adaptive strategy:
```python
F = alpha * (T - t) / T
# Early iterations: Large F (0.9) â†’ strong exploration
# Late iterations: Small F (0.4) â†’ precise exploitation
```

**DE Crossover Rate (CR)** - Based on population diversity:
```python
# Low diversity â†’ High CR (0.9) â†’ more mutation
# High diversity â†’ Low CR (0.5) â†’ preserve structure
```

### Boundary Handling for Discrete Optimization

VM assignments must be integers in range [0, N-1]. Use clipping strategy:
```python
mutant_value = int(np.clip(value, 0, N - 1))
```

## Key Design Decisions

1. **Why 70/30 BCBO/DE split?**
   - BCBO overall performance > MSA (which used 50/50)
   - Preserve BCBO dominance while gaining DE benefits
   - Validated through ablation studies

2. **Why only fuse in first 20% of iterations?**
   - BCBO's attack phases (last 35%) are already highly effective
   - Fusion in weak phases provides better initial solutions for strong phases
   - Reduces computational overhead (+10% vs +30-50% for full fusion)

3. **Why DE over GA/PSO for fusion?**
   - Simpler (only 2 parameters: F, CR vs GA's 4-6 parameters)
   - Better local search precision
   - Proven effective in cloud scheduling (MSA-DE, HDE papers)
   - Differential mutation naturally estimates gradient information
